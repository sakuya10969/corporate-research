# 機能仕様

ユーザーが利用できる機能の一覧と、各機能の振る舞いを定義する。
技術的な実装詳細（エンドポイント定義・スキーマ・エラーコード等）は `api-design.md` を参照。

---

## 実装済み機能（MVP）

### F-001: 企業URL入力

#### 概要

分析したい企業のURLを入力して、分析を開始する。

#### ユーザー操作

1. テキストフィールドに企業のURL（例: `https://www.toyota.co.jp/`）を入力する
2. 「分析する」ボタンを押す
3. 分析が完了するまでローディング表示が出る

#### バリデーション

入力内容が以下を満たさない場合、送信前にエラーメッセージを表示する：

- URLが未入力
- URLの形式が不正
- `http://` または `https://` で始まっていない

#### 制約

- 分析中はフォームが無効化され、二重送信できない
- 入力できるのは企業URLのみ（企業名での検索は非対応）

---

### F-002: 企業分析実行

#### 概要

入力されたURLをもとに、企業の公開情報を自動収集・AI分析して結果を返す。

#### 処理の流れ（ユーザーから見た動作）

1. URLを送信すると、バックグラウンドで企業サイトの情報収集が始まる
2. 収集した情報をAIが構造化・要約する
3. 分析結果がページに表示される

#### 収集・分析される情報

| 項目 | 内容 |
|------|------|
| 企業プロフィール | 社名、設立年、代表者、所在地、従業員数、資本金 |
| 事業領域 | 企業が展開している事業分野 |
| プロダクト・サービス | 提供している製品・サービスの一覧 |
| 財務情報 | 売上高、営業利益、純利益、成長率 |
| ニュース | 最近のプレスリリース・ニュース |
| リスク要因 | 事業上のリスク |
| 企業概要サマリー | AIによる企業の概要説明 |
| 事業モデル | AIによる事業モデルの説明 |
| SWOT分析 | 強み・弱み・機会・脅威 |
| 競合企業（推定） | AIが推定する競合企業 |
| 今後の展望 | AIによる将来展望の考察 |

#### 制約

- 収集対象は入力URLのドメイン内の公開ページのみ（最大15ページ）
- Google検索は使用しない。企業サイトの一次情報のみを使用する
- 情報が取得できなかった項目は表示されない

---

### F-003: 分析結果表示

#### 概要

分析結果をセクションごとに整理して表示する。

#### 表示内容

- 企業名とURLをヘッダーに表示
- 各分析項目をカード形式で表示
  - 企業概要、企業プロフィール、事業モデル、事業領域、プロダクト・サービス
  - 財務情報、SWOT分析、リスク要因、競合企業、今後の展望
  - ニュース、参照ソース
- Markdownレポートを折りたたみで表示（全文確認が可能）
- 参照ソースはカテゴリバッジ付きリンクで表示

#### ローディング・エラー

- 分析中はスケルトンUIを表示する
- エラー発生時はエラーメッセージを表示する
- データが存在しないセクションは非表示になる

---

### F-004: Markdownレポート出力

#### 概要

分析結果をMarkdown形式のレポートとして確認できる。

#### ユーザー操作

- 分析結果ページ下部の「レポート全文を表示」をクリックすると展開される
- テキストをコピーして外部ツールに貼り付けて利用できる

#### レポートに含まれる内容

企業プロフィール、事業領域、プロダクト、財務情報、ニュース、企業概要、事業モデル、SWOT分析、リスク要因、競合企業、今後の展望、参照ソース

## 新規機能（V1）

### F-005: 企業データ永続化

#### 概要

分析した企業の情報をDBに保存し、同じ企業を再入力したときに過去データを活用できるようにする。

#### ユーザーから見た動作

- 同じ企業URLを入力すると、過去に分析済みであることが通知される
- 過去の分析結果をそのまま表示するか、最新情報で更新するかを選択できる
- 初回分析と再分析の両方で、結果はDBに保存される

#### 保存される情報

- 企業の基本情報（URL、社名、正規化済みドメイン）
- 最新の分析結果（構造化データ・要約・ソース一覧）
- 分析実行履歴（いつ・どの種類の分析を実行したか）

#### 設計方針

- Supabase（Postgres）を正本データストアとして使用する
- 構造化データ（structured / summary）はJSONBカラムで保存する
- pgvectorによるembeddingは補助的な検索用途に限定し、業務データの正本はリレーショナルDBで持つ
- 企業は `company_id` で一意に管理し、将来の比較機能に備える

---

### F-006: 差分更新分析

#### 概要

同じ企業を再分析する際、前回から変化のあったページのみを対象に効率的に更新する。

#### ユーザーから見た動作

1. 保存済み企業のURLを入力（または「更新」ボタンを押す）
2. 前回分析からの変更点が差分レポートとして表示される
3. 変更があったセクション（ニュース・IR・採用など）が強調表示される

#### 差分レポートに含まれる内容

- 新しく追加されたニュース・プレスリリース
- 変更された企業プロフィール項目（代表者交代・所在地変更など）
- 新しく追加されたプロダクト・サービス
- 財務情報の変化

#### 変更検知の仕組み

- ページごとにcontent hashを保持し、変化があったページのみ再取得する
- `fetched_at` で前回取得日時を管理する
- HTTPヘッダーにETag / Last-Modifiedがある場合はそれを優先利用する
- ニュース・IR・採用ページは変化頻度が高いため優先的に更新対象とする

#### 制約

- 毎回全ページをフルクロールしない（差分更新型）
- 変更がなかった場合は「変更なし」と表示する

---

### F-007: 深掘り分析

#### 概要

保存済みの企業データをもとに、追加の質問を投げかけて詳細な分析を得られる。

#### ユーザーから見た動作

1. 分析結果ページに「深掘り質問」入力欄が表示される
2. 自由文で質問を入力して送信する
3. AIが保存済みデータをもとに回答する
4. 保存データだけでは不足する場合、自動的に追加収集を行って回答する

#### 質問例

- 「この会社の強みの本質は何か」
- 「競合他社と比べて何が違うか」
- 「成長性の根拠となる情報はあるか」
- 「就活生として注目すべきポイントは何か」

#### 動作の特徴

- 保存済みデータを優先して使用する（毎回フルクロールしない）
- 不足情報がある場合のみ追加収集を行う
- 質問と回答の履歴を会話形式で表示する
- LangGraphによる状態付きワークフローで、途中再開・追加収集を制御する

#### 制約

- 深掘り分析はF-005（企業データ永続化）が前提
- 保存済みデータがない企業には利用不可

---

### F-008: 分析履歴管理

#### 概要

企業ごとの分析実行履歴を一覧で確認できる。

#### ユーザーから見た動作

- 企業詳細ページに「分析履歴」タブが表示される
- 過去の分析実行日時・種別（初回分析 / 差分更新 / 深掘り）が一覧表示される
- 過去の分析結果を選択して内容を確認できる
- 各分析の実行状態（完了・エラー・実行中）が確認できる

#### 履歴に記録される情報

| 項目 | 内容 |
|------|------|
| 実行日時 | 分析を開始した日時 |
| 分析種別 | initial（初回）/ refresh（差分更新）/ deep_research（深掘り） |
| 実行状態 | pending / running / completed / failed |
| 完了日時 | 分析が完了した日時 |
| エラー内容 | 失敗した場合のエラーメッセージ |

---

### F-009: ダウンロード機能

#### 概要

分析結果をファイルとしてダウンロードできる。

#### 対応フォーマット（優先順位順）

| 優先度 | フォーマット | 理由 |
|--------|------------|------|
| 1 | Markdown (.md) | 実装コストが最小。既存のmarkdown_pageをそのまま利用可能 |
| 2 | PDF | 就活生が企業に提出・印刷する用途に対応 |
| 3 | Word (.docx) | 企業研究ノートへの貼り付け・編集用途 |

#### ユーザー操作

1. 分析結果ページの「ダウンロード」ボタンを押す
2. フォーマットを選択する（Markdown / PDF / Word）
3. ファイルがダウンロードされる

#### MVP実装方針

- まずMarkdownダウンロードのみ実装する（既存の `markdown_page` フィールドを利用）
- PDFはV1.1、Wordはその後に追加する

## LangGraphの役割

F-006・F-007で使用するLangGraphは、単発の分析ノード分割ではなく以下の用途に限定して使用する：

- 保存済み企業データを前提とした再分析・差分更新
- 深掘り質問時の状態付きワークフロー（途中再開・追加収集の制御）
- チェックポイントを活かした長時間処理の管理

毎回フルスクレイピング → フル再分析するだけの用途には使用しない。

---

## アーキテクチャ方針

### データ永続化

- Supabase（Postgres）を正本データストアとして使用する
- 構造化データはJSONBで保存する
- pgvectorによるembeddingは補助的な検索用途のみ（業務データの正本はリレーショナルDBで持つ）
- 企業は `company_id` で一意に管理する

### 避けるべき設計

- 毎回URLを入力するたびにフルスクレイピング → フル再分析する設計
- ベクトルDB / embedding側だけに企業の正本状態を持たせる設計
- 分析結果を毎回RAGで再生成し、永続化しない設計

## 新規機能（V2）

### F-010: 分析テンプレート（目的別プリセット）

#### 概要

分析の目的に応じたプリセットを選択することで、重点的に分析する観点を変えられる。

#### ユーザーから見た動作

1. 企業URL入力フォームの下に「分析の目的」セレクターが表示される
2. 目的を選択して分析を実行する
3. 選択した目的に応じた観点で分析結果が強調・整理される

#### テンプレート一覧

| テンプレートID | 名前 | 主な用途 | 重点観点 |
|--------------|------|---------|---------|
| `general` | 総合分析（デフォルト） | 汎用 | 全項目均等 |
| `job_hunting` | 就活・転職リサーチ | 就活生・転職者 | 企業文化・採用情報・成長性・安定性 |
| `investment` | 投資リサーチ | 個人投資家・アナリスト | 財務情報・成長率・競合優位性・リスク |
| `competitor` | 競合調査 | 事業開発・営業 | 事業領域・プロダクト・差別化要因・市場ポジション |
| `partnership` | 提携先調査 | BD・アライアンス担当 | 事業シナジー・技術力・財務健全性 |

#### テンプレートによる出力の違い

- `job_hunting`: 採用情報・社風・福利厚生・成長機会を重点抽出。「就活生へのアドバイス」セクションを追加
- `investment`: 財務指標・成長率・競合比較・リスク要因を重点抽出。投資判断サマリーを追加
- `competitor`: 事業領域・プロダクト・差別化要因・市場ポジションを重点抽出。競合マップを追加
- `partnership`: 技術力・事業シナジー・財務健全性を重点抽出。提携可能性サマリーを追加

#### 制約

- テンプレートはプロンプトレベルで制御する（スキーマ変更なし）
- `general` がデフォルト。未選択時は `general` として処理する

### F-011: 企業名検索（URL補完）

#### 概要

企業URLを知らなくても、企業名を入力するだけで公式サイトURLを自動補完できる。

#### ユーザーから見た動作

1. 入力フォームに企業名（例: 「トヨタ」「Sony」）を入力する
2. 候補となる企業と公式URLが最大5件表示される
3. 候補を選択すると、URLフィールドに自動入力される
4. そのまま「分析する」ボタンを押して分析を開始できる

#### 検索の仕組み

- DuckDuckGo Instant Answer API（無料・APIキー不要）を使用して企業の公式サイトURLを取得する
- 取得したURLはフロントエンドに返却し、ユーザーが確認・選択する
- 自動的に分析を開始しない（ユーザーが確認してから実行）

#### バリデーション

- 企業名が1文字以上であること
- 検索結果が0件の場合は「見つかりませんでした。URLを直接入力してください」と表示する

#### 制約

- 検索結果の精度はDuckDuckGo依存。公式サイト以外が候補に含まれる場合がある
- ユーザーが候補を確認・選択する手順を必ず挟む（自動実行しない）

---

### F-012: 分析結果シェア

#### 概要

分析結果を固有URLで共有できる。URLを知っている人なら誰でも閲覧できる。

#### ユーザーから見た動作

1. 分析結果ページに「シェア」ボタンが表示される
2. ボタンを押すと共有用URLがクリップボードにコピーされる（例: `https://app.example.com/share/abc123`）
3. URLを受け取った人はブラウザで開くと分析結果を閲覧できる
4. 共有ページには「この分析結果を元に自分でも分析する」ボタンを表示する

#### 共有ページの仕様

- 分析結果の全セクションを閲覧できる（読み取り専用）
- 分析実行日時を表示する
- OGP（Open Graph Protocol）メタタグを設定し、SNSシェア時にプレビューが表示される
  - タイトル: `{企業名} の企業分析レポート`
  - 説明: overview の先頭150文字

#### 前提条件

- F-005（企業データ永続化）が完成していること
- 共有IDは `analysis_id` をベースにした短縮ID（UUID v4 の先頭8文字）

#### 制約

- 共有URLは認証不要で閲覧可能（パブリックリンク）
- 共有URLの有効期限はMVPでは設定しない（永続）
- 削除機能はV2.1以降に追加する

---

### F-013: 複数企業比較

#### 概要

2〜3社の企業を並べて比較分析できる。

#### ユーザーから見た動作

1. トップページに「比較分析」モードへの切り替えボタンを表示する
2. 比較モードでは企業URL入力フィールドが最大3つ表示される
3. 「比較分析する」ボタンを押すと、各企業の分析を並行実行する
4. 結果は横並びの比較テーブルで表示される

#### 比較表示の内容

| 比較項目 | 表示形式 |
|---------|---------|
| 企業プロフィール（社名・設立・従業員数・資本金） | テーブル横並び |
| 事業領域 | タグ一覧（各社色分け） |
| 財務情報（売上・利益・成長率） | バーチャートまたはテーブル |
| SWOT分析 | 各社のSWOTカード横並び |
| 競合優位性サマリー | AIによる比較コメント |
| 総合スコア（F-014連携） | レーダーチャート |

#### AIによる比較コメント

- 各企業の分析結果をまとめてLLMに渡し、「3社の比較サマリー」を生成する
- 「どの企業が〇〇の観点で優れているか」を自然文で説明する

#### 前提条件

- F-005（企業データ永続化）が完成していること（保存済みデータを再利用）
- 保存済みでない企業は新規分析を実行してから比較する

#### 制約

- 比較できるのは最大3社まで
- 各社の分析は並行実行する（逐次実行しない）
- 比較結果はURLパラメータで共有可能（例: `?compare=url1,url2,url3`）

### F-014: 分析スコアリング

#### 概要

分析結果を複数の観点で数値スコア化し、企業の強み・弱みを直感的に把握できる。

#### ユーザーから見た動作

- 分析結果ページの上部に「スコアサマリー」カードが表示される
- 各観点のスコア（0〜100点）とレーダーチャートが表示される
- スコアの根拠となる情報が各項目にツールチップで表示される

#### スコア観点

| 観点 | 説明 | 主な根拠データ |
|------|------|--------------|
| 財務健全性 | 売上・利益・成長率の安定度 | financials |
| 成長性 | 事業拡大・新規領域への展開度 | outlook, news |
| 競合優位性 | 差別化要因・市場ポジション | swot.strengths, competitors |
| リスク度 | 事業リスクの多さ・深刻度 | risks |
| 情報透明性 | 公開情報の充実度 | sources の数・カテゴリ多様性 |

#### スコア算出の仕組み

- LLMがSummaryDataとStructuredDataをもとに各観点を0〜100でスコアリングする
- スコアの根拠となる1〜2文の説明文も同時に生成する
- スコアはAnalysisResponseに追加フィールドとして含める

#### 制約

- スコアはLLMによる推定値であり、客観的な評価ではない旨を明示する
- スコアの算出はStage 2（要約生成）と同じLLM呼び出しで行う（追加コストを最小化）
